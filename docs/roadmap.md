# OpenYield Project Roadmap

**Author:** Yeonkuk Woo
**Last Updated:** June 2025

This roadmap defines the named deliverables for OpenYield across three development phases. Phase 1 is complete. Phases 2 and 3 are scoped based on the technical gaps identified during Phase 1 development and the roadmap requirements of the primary beneficiary categories (domestic glass substrate manufacturers, silicon wafer fabs, national laboratories, academic ML researchers, SEMI consortium, defense and aerospace electronics producers).

## Phase 2 — Advanced Defect Inspection Capabilities (2025–2026)

**Target:** Q4 2025 – Q2 2026

Phase 2 extends the AI and visualization capabilities and introduces critical area extraction for advanced-node yield modeling.

### 2.1 Convolutional Neural Network Defect Classifier

**Deliverable:** `ai/cnn_classifier.py` — image-based defect classifier trained on the 64×64 PNG patches generated by `synthetic/image_generator.py`.

**Technical scope:**
- Minimal CNN architecture (2–3 convolutional layers + global average pooling + softmax) implementable in NumPy or introduced as an optional PyTorch dependency under `[project.optional-dependencies.ai]`
- Training pipeline that consumes `defect_images` records and `defect_predictions` ground truth labels
- Model persistence to `model_registry` alongside the Phase 1 logistic regression model
- Comparison report against the Phase 1 classifier baseline

**Beneficiary impact:** Academic ML researchers can benchmark CNN architectures against the logistic regression baseline on the same synthetic dataset. Domestic fabs can evaluate image-based classification accuracy on their real defect images.

### 2.2 Wafer Map Visualization API

**Deliverable:** `/panels/{panel_id}/map` endpoint returning a spatial defect density representation suitable for rendering as a heatmap.

**Technical scope:**
- Grid-based defect density aggregation (configurable bin size)
- JSON response format compatible with standard heatmap visualization libraries
- Optional cluster overlay (DBSCAN cluster assignments per bin)
- Yield model gradient overlay (per-bin expected yield contribution)

**Beneficiary impact:** Fab engineers can visualize defect spatial patterns directly from the API without requiring a separate visualization tool.

### 2.3 Critical Area Extraction (Basic)

**Deliverable:** `yield_engine/critical_area.py` — extracts the critical area fraction from the component grid and defect size distribution.

**Technical scope:**
- Geometric critical area model: probability that a defect of size `s` at position `(x, y)` kills a die, based on die critical feature dimensions
- Integration with `yield_engine/calculator.py` to replace `A = full_die_area` with `A = critical_area_fraction × full_die_area`
- Substrate-profile-defined default critical area fractions for each substrate type
- Backward-compatible: existing yield estimates computed with `full_die_area` are preserved in the database

**Beneficiary impact:** Silicon wafer fabs at advanced nodes (<28nm) require critical area–corrected yield estimates. Without critical area extraction, yield estimates at these nodes can be off by a factor of 2–5×.

### 2.4 Multi-Lot Trend Analysis

**Deliverable:** `analysis/trend.py` — tracks yield and defect density trends across lots in chronological order.

**Technical scope:**
- Rolling mean and standard deviation of defect density across sequential lots
- Process drift detection: statistically significant monotonic trend in defect density
- Correlation between lot-level defect density and yield (Pearson r with p-value)
- REST API endpoint: `GET /lots/trend`

**Beneficiary impact:** Lot-to-lot trend analysis is the primary tool for detecting slow process drift before it causes a yield excursion. This capability is standard in enterprise yield management platforms but requires lot chronology data that must be tracked across the `lot_summaries` table.

### 2.5 Ruff Linting CI Integration

**Deliverable:** `.github/workflows/ci.yml` — GitHub Actions workflow running `pytest` and `ruff check` on every pull request.

**Technical scope:** Standard GitHub Actions workflow targeting Python 3.11 and 3.12. No new runtime dependencies.

## Phase 3 — Integration and Ecosystem Development (2026–2027)

**Target:** Q3 2026 – Q1 2027

Phase 3 establishes OpenYield as a connectable component within the broader open-source manufacturing software ecosystem.

### 3.1 Spatial Yield Prediction

**Deliverable:** `yield_engine/spatial_predictor.py` — integrates yield models over a spatial defect density map to produce a within-wafer yield estimate with spatial resolution.

**Technical scope:**
- Divide the wafer/panel into spatial bins
- Compute per-bin defect density from ingested defect records
- Apply the appropriate yield model to each bin
- Aggregate bin yields to produce a panel yield estimate that accounts for within-wafer non-uniformity
- REST API endpoint: `POST /yield/{panel_id}/spatial`

**Beneficiary impact:** This is the approach used by KLA Klarity's yield prediction module. Implementing it in OpenYield eliminates one of the last technical gaps between the open-source platform and enterprise capabilities.

### 3.2 KLARF 2.0 Binary Format Support

**Deliverable:** `ingestion/adapters/klarf2_adapter.py` — parser for KLARF 2.0 binary format.

**Technical scope:**
- Pure-Python binary parser for the KLARF 2.0 format
- Produces the same `NormalizedDefect` records as the KLARF 1.x adapter
- Backward-compatible: the existing KLARF 1.x adapter is unchanged

**Beneficiary impact:** KLARF 2.0 adoption is increasing in new tool installations. Supporting both format versions ensures OpenYield remains compatible with the full installed base of domestic inspection tools.

### 3.3 Lot Genealogy Tracking

**Deliverable:** `analysis/genealogy.py` — tracks equipment-to-defect attribution by correlating defect spatial patterns with equipment process steps.

**Technical scope:**
- Equipment event log ingestion (CSV format)
- Spatial correlation between defect cluster onset and equipment events
- Attribution score: probability that a specific equipment event caused a specific cluster
- REST API endpoints: `POST /lots/{lot_id}/genealogy`, `GET /lots/{lot_id}/attribution`

**Beneficiary impact:** Root cause attribution — determining which piece of equipment caused a yield excursion — is the highest-value capability in a yield management platform. This feature moves OpenYield from defect reporting to process diagnosis.

### 3.4 OpenMES Integration Connector

**Deliverable:** `integrations/openmes_connector.py` — bidirectional data exchange connector for the OpenMES open-source manufacturing execution system.

**Technical scope:**
- Pulls work order and lot routing data from OpenMES to populate `lots` table
- Pushes yield estimates and excursion alerts back to OpenMES as process events
- Authentication via API key (configurable)

**Beneficiary impact:** Integrating yield management data with the manufacturing execution system closes the loop between process results and production decisions. This integration is required for domestic fabs building fully open-source manufacturing software stacks.

### 3.5 SEMI Standards Compliance Documentation

**Deliverable:** `docs/semi-compliance.md` — mapping of OpenYield data structures and API endpoints to relevant SEMI standards.

**Technical scope:**
- SEMI E10 (Equipment Reliability, Availability, and Maintainability)
- SEMI E30 (Generic Model for Communications and Control of Manufacturing Equipment)
- SEMI E142 (Substrate Mapping)
- SEMI E179 (Guide for Defect Identification)

**Beneficiary impact:** SEMI consortium members evaluating OpenYield for interoperability testing require a standards compliance mapping. This document makes the compliance assessment tractable.

## Contribution to the CHIPS Act Objective

Each phase of this roadmap directly addresses capabilities required by the primary beneficiary categories identified in the CHIPS Act:

- **Phase 1** establishes the open data infrastructure foundation — ingestion, validation, yield modeling — that any domestic fab needs to operate.
- **Phase 2** brings analytical depth — image-based ML, critical area extraction, process drift — that closes the gap between OpenYield and enterprise tools for advanced-node fabs.
- **Phase 3** establishes integration and ecosystem connectivity — lot genealogy, MES connectors, standards compliance — that enables OpenYield to function as a component in a fully open-source manufacturing software stack for domestic facilities.

The completion of all three phases would produce an open-source yield management platform comparable in analytical capability to KLA Klarity, available at zero licensing cost to every CHIPS Act beneficiary.
